## Контекст

- **v1.X** — стабильная линия с **API v2**, админ-панелью, сессиями в БД и экспортом отчётов. Ориентир: `v1.0.0`.
- **v2.X** — релиз `v2.0.0` с **API v3**. Фокус: безопасность, изоляция данных, производительность, эксплуатация.

HTTP-префикс по-прежнему `/api`, формат `{ success, data, message, code }` сохранён. Breaking-изменения API — в [API v2 vs v3](/wiki/api-v2-v3).

## Что характерно для v1.X

В v1.X уже были заложены:

- единый слой ответов API (`successResponse`, `handleApiError`, `DB_OFFLINE`);
- регистрация через заявку (`pending` → `approved` / `rejected`);
- JWT-сессии с хранением в БД и отзывом;
- админ-панель, аудит, управление группами и пользователями;
- CSRF на мутациях и rate limit на части auth/API-маршрутов;
- экспорт DOCX, wiki, разделение UI по вкладкам группы.

Ограничения, которые проявлялись на практике:

- вложенные API группы (attendance, grades, students) проверяли только факт входа, **не владельца группы**;
- `GET /groups` возвращал **все** группы любому авторизованному пользователю;
- защита `/profile` и `/admin` опиралась на **клиентский редирект** (`getSession` + `router.push`);
- разные тексты ошибок логина («пользователь не найден» / «неверный пароль») упрощали перебор email;
- Server Actions ходили в свой API по HTTP (лишний round-trip);
- схема БД создавалась одним `CREATE IF NOT EXISTS` без нумерованных миграций;
- посещаемость и оценки жёстко привязаны к строке `full_name`, без `fk_student`;
- не было health-check, CI, автотестов и скрипта резервного копирования SQLite;
- rate limiter хранил данные в памяти процесса — при перезапуске лимиты сбрасывались;
- логи писались только в `console.*` и терялись при перезапуске;
- операции со студентами выполнялись по одному запросу в цикле (N+1);
- `last_seen_at` обновлялся при каждом запросе — лишняя запись в БД;
- монолитный `functions.ts` (600+ строк) содержал несвязанные утилиты;
- `handleApiResponse` дублировался в двух файлах с разным поведением;
- SQLite-обёртки были помечены `async` без необходимости;
- ключи из объекта данных подставлялись в SQL без фильтрации — потенциальная SQL-инъекция;
- параметр `redirectTo` при логине/регистрации не валидировался — возможна open redirect атака.

## Что меняется в v2.X

### Безопасность и доступ

| Область | v1.X | v2.X |
| --- | --- | --- |
| Вложенные API группы | Любой залогиненный пользователь по ID группы | Проверка владельца (`fk_user`) или `canAccessAdmin` |
| `GET /groups` | Все группы | Только свои; админы видят все |
| Страницы `/profile`, `/admin` | Клиентский редирект | `middleware.ts` — проверка JWT в cookie до отдачи страницы |
| Сообщения при логине | Разные для «нет email» и «неверный пароль» | Единое: «Неверный email или пароль» |
| Регистрация (enumeration) | «Пользователь уже существует» | Нейтральный успех, как при новой заявке |
| Rate limit на вход | Только `POST /api/auth/login` | Также Server Action `Login` и `Register` (формы на `/login` и `/register`) |
| Подтверждение пароля | — | Поле «Подтвердите пароль» в форме регистрации |
| CSP | `unsafe-eval` + `unsafe-inline` | `unsafe-eval` **убран** из `script-src` |
| CSRF-токен | maxAge 24 часа | maxAge сокращён до **1 часа** |
| SQL-инъекция | Ключи объекта подставлялись в SQL | Whitelist допустимых колонок |
| Open redirect | Нет валидации `redirectTo` | Проверка принадлежности к текущему домену |
| Одна группа на преподавателя | Задумано, не enforced | `400` при повторном создании |

### Архитектура и данные

| Область | v1.X | v2.X |
| --- | --- | --- |
| Server Actions | HTTP → `/api/...` | Сервисный слой `groups-service.ts` → SQLite напрямую |
| БД | SQLite, одна схема при старте | Таблица `schema_migrations`, миграции 1–4 (индексы, `fk_student`, backfill) |
| Связь attendance/grades | Только `full_name` | Колонка `fk_student` + сохранение при upsert |
| Сессии админа (SQL) | `LIMIT` в строке запроса | Параметризованные плейсхолдеры |
| Валидация | Дубли в admin page / API | Общий `utils/validation.ts` |
| Файл-утилита | Монолитный `functions.ts` (600+ строк) | Разделён на `classnames.ts`, `api-errors.ts`, `functions.ts` (только Word-экспорт) |
| SQLite-обёртки | `async` с лишним Promise overhead | Синхронные (`query`, `queryOne`, `execute`, `transaction`) |
| Типизация handlers | `data: object` | Типизированные интерфейсы (`UpdateGroupData`, `UpdateProfileFormData`) |
| `handleApiResponse` | Дублирование в `api.ts` и `functions.ts` | Единая реализация в `api-errors.ts` |

### Эксплуатация и качество

| Область | v1.X | v2.X |
| --- | --- | --- |
| Health | — | `GET /api/health` (версия, статус БД) |
| Резервное копирование | — | `npm run backup:db` (WAL checkpoint + backup) |
| CI | — | GitHub Actions: typecheck, lint, test, build |
| Тесты | — | Vitest (`npm run test`), 13 тестов валидации |
| Typecheck | — | `npm run typecheck` |
| Rate limit | In-memory (теряется при рестарте) | **SQLite-хранилище** — данные сохраняются при перезапуске |
| Логирование | Только `console.*` | Консоль + файл `logs/app.log` |
| Batch-операции | По одному INSERT/UPDATE на студента | Транзакции с prepared statements |
| `last_seen_at` | UPDATE на каждый запрос | Обновление раз в **5 минут** |

### UX и интерфейс

| Область | v1.X | v2.X |
| --- | --- | --- |
| Вкладки группы | Все три монтируются сразу | Lazy load: dynamic import + загрузка данных при первом открытии вкладки |
| ARIA / клавиатура | Минимально | `tablist` / `tab` / `tabpanel`, стрелки Home/End на группе и в профиле |
| Плейсholders login | `test@test.com` | Нейтральный `name@example.com` |
| Типизация отчётов | `any` в ReportEditDialog | `GradesReportData`, `AttendanceReportData` |
| Комментарии в коде | Тривиальные комментарии к каждой строке | Убраны избыточные комментарии, оставлены только для неочевидной логики |

## Сводное сравнение по фокусу

| | v1.X | v2.X |
| --- | --- | --- |
| Продуктовый фокус | Стабильный MVP + API v2 | Hardening + производительность + **API v3** |
| Модель API | v2 (слабая изоляция групп) | **v3** (403 на чужие группы) |
| Модель угроз | «Доверяем UI» | «Не доверяем клиенту» — права на API и middleware |
| Интеграции | Bearer + cookie, единый JSON | То же + строже 403 на чужие группы |
| Миграция данных | — | Авто-миграции при старте приложения |
| Rate limit | In-memory | SQLite-персистентный |
| Логирование | Консоль | Консоль + файл |
| Документация | v0 vs v1, API v1 vs v2 | + эта страница (v1 vs v2 продукта) |

## Миграция с v1.X на v2.X

### Для преподавателей (UI)

Поведение интерфейса **почти не меняется**: те же вкладки, те же формы. Отличия:

- чужие группы **не видны** в списке и **недоступны** по прямой ссылке;
- при истёкшей сессии редирект на `/login` происходит **раньше** (middleware);
- у преподавателя может быть **только одна** закреплённая группа;
- при регистрации нужно **подтвердить пароль** (второе поле).

### Для интеграторов (API)

Полный список отличий: **[API v2 vs v3](/wiki/api-v2-v3)**. Кратко:

1. **Не полагайтесь на `GET /groups` для «всех групп»** — ответ фильтруется по текущему пользователю.
2. Ожидайте **`403 FORBIDDEN`** на attendance/grades/students/stats чужой группы.
3. Логин: при неверных данных **`401`** с текстом «Неверный email или пароль».
4. Register: при занятом email — тот же **success**, что и для новой заявки.
5. Мониторинг: **`GET /api/health`**.
6. Мутации — **CSRF** без изменений (`GET /api/auth/csrf` + `x-csrf-token`), но токен живёт **1 час** (не 24).

### Для администраторов

- Админы с `canAccessAdmin` по-прежнему видят все группы и проходят проверку доступа к любой группе.
- Рекомендуется настроить **регулярный backup**: `npm run backup:db`.
- Перед деплоем v2 проверьте, что на сервере выполняются миграции (первый запуск после обновления создаст `schema_migrations` и применит недостающие шаги).
- Логи пишутся в `logs/app.log` — настройте ротацию или мониторинг.

## Roadmap v2 (после релиза)

Запланировано для последующих обновлений:

- распределённый rate limit (Redis / Upstash) для кластерного деплоя;
- ужесточение CSP без `unsafe-inline` (nonce-based подход);
- полный переход attendance/grades на `fk_student` как единственный ключ связи;
- расширенный набор тестов (E2E, права, CSRF, rate limit);
- пагинация в списках групп и пользователей;
- поиск и фильтрация в таблицах;
- мобильная адаптация широких таблиц;
- предупреждение о несохранённых данных при навигации;
- email/Telegram-уведомления админу о новых заявках;
- аудит изменений посещаемости/оценок преподавателем.
