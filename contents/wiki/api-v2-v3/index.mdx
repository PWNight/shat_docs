## Линейка поколений API

| Поколение | Продукт | Префикс | Статус |
| --- | --- | --- | --- |
| **API v1** | ранние сборки | `/api/...` | legacy, снято с поддержки |
| **API v2** | **v1.X** (`v1.0.0`) | `/api/...` | предыдущее стабильное поколение |
| **API v3** | **v2.X** (`2.0.0-beta1+`) | `/api/...` | **текущее**, см. [API Reference](/wiki/api-reference) |

URL-префикс **не менялся** — по-прежнему `/api`. Меняются **контракты отдельных операций**, коды ошибок и модель доступа. Формат обёртки `{ success, data, message, code }` сохранён.

Историческое сравнение v1 → v2: [API v1 vs v2](/wiki/api-v1-v2).  
Сравнение продуктовых релизов: [Версии v1.X vs v2.X](/wiki/versions-v1-v2).

---

## Что общего (v2 и v3)

Эти возможности появились ещё в **API v2** и в **v3** не убраны:

- единые утилиты ответов (`successResponse`, `handleApiError`, `DB_OFFLINE` → `503`);
- авторизация: cookie `session` + `Authorization: Bearer <token>`;
- регистрация как заявка (`pending` / `approved` / `rejected`);
- CSRF на мутациях (`GET /api/auth/csrf`, заголовок `x-csrf-token`);
- rate limit на auth и большинстве маршрутов (`429`, `Retry-After`);
- Zod-валидация через `validateData`, безопасный `safeParseJson`;
- admin-раздел `/api/admin/*`, сессии, password-reset, stats группы;
- те же пути: `/groups`, `/groups/{id}/attendance`, `/grades`, `/students` и т.д.

## Главные отличия v3 от v2

### 1) Модель доступа к группам (breaking)

В **API v2** для вложенных ресурсов группы достаточно было **любой** валидной сессии, зная `id` группы.

В **API v3** действует **изоляция по владельцу**:

| Операция | API v2 | API v3 |
| --- | --- | --- |
| `GET /groups` | Все группы в системе | Только группы текущего пользователя; `canAccessAdmin` — все |
| `GET /groups/{id}` | Любой авторизованный | Владелец или админ, иначе **403** |
| `GET/PATCH/DELETE /groups/{id}` (PATCH/DELETE) | Владелец (PATCH/DELETE) | Без изменений для PATCH/DELETE; GET — как выше |
| `.../attendance`, `.../grades`, `.../students`, `.../stats` | Любой авторизованный | Владелец или админ, иначе **403** |
| `POST .../attendance`, `.../grades` (body `groupId`) | Без проверки владельца body | Проверка доступа к `groupId` |

Типичный новый ответ v3:

```json
{
  "success": false,
  "message": "Нет доступа к этой группе",
  "code": "FORBIDDEN"
}
```

HTTP **403**.

**Миграция:** не используйте `GET /groups` как «список всех групп в техникуме». Для админ-сценариев нужен пользователь с `canAccessAdmin` или admin API.

---

### 2) Создание группы — одна на преподавателя

| | API v2 | API v3 |
| --- | --- | --- |
| `POST /groups` | Повторное создание для того же `fk_user` не блокировалось | **400** — «У преподавателя уже есть закреплённая группа» |

---

### 3) Auth: сообщения и перебор учёток

| Сценарий | API v2 | API v3 |
| --- | --- | --- |
| Неверный email | «Пользователь с такой почтой не найден» | «**Неверный email или пароль**» |
| Неверный пароль | «Неправильный пароль» | «**Неверный email или пароль**» |
| Pending после верного пароля | Сообщение о pending | Без изменений |
| `POST /auth/register`, email занят | **401** / явная ошибка «уже существует» | **200** + то же success-сообщение, что и при новой заявке |

Клиентам v3 **нельзя** определять существование email по тексту login/register.

---

### 4) Новые и служебные эндпоинты

| Эндпоинт | API v2 | API v3 |
| --- | --- | --- |
| `GET /api/health` | — | **200** / **503**, версия приложения, `database: connected \| offline` |

Пример v3:

```json
{
  "success": true,
  "data": {
    "status": "ok",
    "version": "2.0.0-beta1",
    "database": "connected",
    "timestamp": "2026-05-18T12:00:00.000Z"
  }
}
```

---

### 5) Данные attendance / grades

| | API v2 | API v3 |
| --- | --- | --- |
| Связь со студентом | Только поле `full_name` в JSON | Дополнительно пишется `fk_student` в БД (если студент найден в группе) |
| Формат API-запросов | `fullName` в теле | **Без изменений** для клиента — те же поля |

Обратная совместимость тел запросов сохранена; меняется только внутреннее хранение.

---

### 6) Rate limit

| | API v2 | API v3 |
| --- | --- | --- |
| `POST /auth/login` | 5 / 15 мин на IP | Без изменений |
| `POST /auth/register` | 3 / час | Без изменений |
| Per-user лимиты на groups/attendance/… | Есть | Без изменений |
| Хранилище лимитов | In-memory процесса | In-memory (Redis — в roadmap) |

## Сводная таблица по эндпоинтам

| Эндпоинт | Изменение в v3 |
| --- | --- |
| `GET /health` | **Новый** |
| `POST /auth/login` | Другие тексты **401** |
| `POST /auth/register` | Нет **401** «email занят»; всегда success-message |
| `GET /groups` | **Фильтр** по текущему пользователю |
| `GET /groups/{id}` | **403** для чужих |
| `POST /groups` | **400** при второй группе преподавателя |
| `PATCH/DELETE /groups/{id}` | Логика владельца как в v2 |
| `GET/POST/DELETE .../attendance` | **403** без прав на группу |
| `GET/POST/DELETE .../grades` | **403** без прав на группу |
| `GET/POST .../students` | **403** без прав на группу |
| `PATCH/DELETE .../students/{id}` | **403** без прав на группу |
| `GET .../stats` | **403** без прав на группу |
| `/admin/*`, `/users/*`, `/sessions/*` | Контракты **без breaking-изменений** |
| `GET /auth/csrf` | Без изменений |

## Чеклист миграции v2 → v3

1. **Список групп** — ожидайте только «свои» записи; не парсьте чужие `id` из старых дампов.
2. **Прямые ссылки** на `/groups/{id}/...` — обрабатывайте **403**, не только 404.
3. **Login UI** — одно сообщение об ошибке credentials; не ищите подстроки «не найден» / «пароль».
4. **Register** — не трактуйте success как «новый пользователь»; email мог уже существовать.
5. **Мониторинг** — добавьте опрос `GET /api/health`.
6. **Админ-интеграции** — для кросс-групповых операций используйте учётку с `canAccessAdmin` или `/api/admin/*`.
7. **Тесты** — покройте сценарий «пользователь A не читает группу B» (regression v2).

## Совместимость версий

| Клиент написан под | Работа с v3 без правок |
| --- | --- |
| API v1 | **Нет** (см. [v1 vs v2](/wiki/api-v1-v2)) |
| API v2, только свои группы | **Частично** — тексты auth и `POST /groups` |
| API v2, опирался на все группы в `GET /groups` | **Нет** — нужна доработка |
| API v2, перебор чужих `groupId` | **Нет** — получите 403 |
